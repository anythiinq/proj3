# report

- A description of the control flow for the interaction of a lemming and a trampoline. Where in the code is the co-location of the two objects detected, and what happens from that point until the interaction is finished? Which functions of which objects are called and what do they do during the handling of this situation?
    - `Trampoline::doSomething()` calls a function through the pointer back to `StudentWorld` using `world()`. The function in class `StudentWorld` is called `getBounceableLemmingAt(Coord c)`, which returns a pointer to a lemming that passes all the conditions required to bounce on the trampoline. In that function, the world iterates through `m_actors`, and if an actor is bounceable, then `Coord c` is compared to the coordinates of the individual actors in `m_actors` to detect co-location.
    - If the coordinates are the same, then that actor is checked to see if it can be launched using the function `canBeLaunched()`. `canBeLaunched()` is a `virtual` function in class `Actor` and only returns true if the actor is a `Lemming` and it is not already in its bouncing state. This prevents a lemming from being relaunched while mid-bounce. If `canBeLaunched()` returns true, then the pointer to that lemming object is returned to `Trampoline::doSomething()`.
    - If the pointer is not `nullptr`, then the bounce will occur. The lemming pointer then calls the public function `launch()`. `launch()` begins by calculating the target bounce distance (based on the lemming’s current fall distance), and then initializing the `targetBounceDistance` and `upwardStepsAttempted` variables. Finally, it sets the lemming’s `m_state` to `BOUNCING`, which allows the bouncing behavior to be processed during subsequent ticks inside `Lemming::doSomething()`.
